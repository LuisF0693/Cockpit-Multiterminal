# Story 16.3: Adoção ao vivo de sessões externas do daemon

## Status

InReview

## Story

**As a** agente orquestrador (Orion na sessão master),
**I want** que sessões criadas por clientes externos do daemon (via named pipe) virem tiles no canvas na hora,
**so that** eu abra terminais novos com a IA certa pra cada tarefa sem o fundador relançar o app.

## Acceptance Criteria

1. Planner puro `planExternalAdoption` (core, sem I/O — mesmo princípio da 9.2): recebe sessões vivas do daemon + ids conhecidos + projetos; devolve planos de adoção com `projectId` derivado do rootPath MAIS específico que contém o cwd (case-insensitive, separadores mistos, boundary de segmento).
2. Main faz poll leve (4s, `unref`) após o resume do boot: sessões desconhecidas são adotadas (`adoptPty` + `registry.adopt`) — o evento `created` já existente cria o tile no renderer (`upsertSession` + `addTile`) e a porta é entregue mid-run via `postMessage`.
3. Race com creates do próprio app fechado: `knownIds` inclui `parkedPorts` (o id entra lá ANTES de `registry.create` resolver).
4. Sessão adotada nasce com nome `{adapterId} (externo)`, workspace ativo, e entra na persistência pelo caminho existente (re-upsert do `created`, 6.3) — sobrevive a restart como qualquer terminal.

## Dev Notes

- O daemon já sustenta tudo: `create` de qualquer cliente do pipe, sessão sobrevive à desconexão do criador ("desconexão NÃO dispõe sessões", 6.1), `configure` é server-wide (o app configura no boot).
- `adoptFromDaemon` (6.3) continua intocado — ele cobre o boot (sessões persistidas); esta story cobre o runtime (sessões desconhecidas). Poll só liga após `performResume` pra nunca competir com o boot.
- Guard `adoptingExternal` evita reentrância se um tick atrasar (adoptPty é async).
- Ferramenta operacional do orquestrador (fora do produto): `cockpit-bridge.js` (scratchpad do Orion) — `list`, `send <id> <texto>`, `create <adapterId> [cwd]`.
