# Story 18.4: Histórico de despachos

## Status

Done

## Story

**As a** chefe que despacha workers pelo Cockpit,
**I want** um registro histórico de quem despachou quem, com que adapter/modelo e qual foi o desfecho,
**so that** eu (ou uma feature futura, como o FR63) tenha dado factual em vez de achismo pra decisões de roteamento.

## Acceptance Criteria

1. Nova entidade `DispatchRecord` — mesmo padrão arquitetural de `TaskManager`/`LearningManager` (estado vivo em `Map` + persistência via `WriteQueue`, sem tabela de junção nova além da própria).
2. Um registro é criado no MOMENTO em que `dispatchedBy` é resolvido na adoção externa (mesmo ponto do vínculo automático, Story 17.2) — campos: `dispatchedBy` (chefe), `label` (agente), `adapterId`, `model` (se houver, extraído de `plan.args`/equivalente), `projectId`, `createdAt`.
3. Quando o worker atinge um estado terminal (`done`, `error`, ou fecha), o registro é atualizado com o desfecho — reusa o listener de status já existente (mesmo ponto que dispara `planTerminalLinkRouting`).
4. Sem vínculo detectado (`dispatchedBy` null), NENHUM registro é criado — histórico é só pra despachos rastreáveis.
5. API/IPC de consulta simples (lista, sem paginação nesta story) — painel de UI fica FORA de escopo (é uma ideia separada, "árvore de delegação", não formalizada ainda).

## Dev Agent Record

### Design da entidade

`DispatchRecord` (`PersistedDispatchRecord` no state-store) segue EXATAMENTE o
padrão do `LearningManager`/`TaskManager`: estado vivo numa `Map` no
`DispatchManager`, persistência assíncrona via `WriteQueue` injetado (o
chamador nunca espera I/O). Campos: `id`, `dispatchedBy` (chefe), `workerId`
(worker), `label` (agente, ex.: "@dev"), `adapterId`, `model` (nullable),
`projectId` (nullable), `createdAt`, `outcome` (nullable — `'done' | 'error' |
'closed'`), `outcomeAt` (nullable). Tabela nova `dispatch_records` via `CREATE
TABLE IF NOT EXISTS` (mesmo caso de `learnings`/`terminal_links`/
`browser_tiles` — nenhuma migração `ALTER` necessária); `SCHEMA_VERSION`
bumped de `'10'` para `'11'`. Índice `idx_dispatch_records_worker` acelera o
`UPDATE ... WHERE worker_id = ?` de `recordOutcome`. `DispatchManager` mantém
um índice auxiliar `byWorkerId: Map<workerId, recordId>` pra que
`recordOutcome` nunca precise varrer a `Map` inteira.

### Decisões tomadas

1. **`model` fica sempre `null` na criação.** Confirmado nos Dev Notes da
   story: o Épico 17.3 escolhe modelo por sessão via `--model`, mas essa
   escolha não é propagada até o ponto de adoção externa (`ExternalAdoptionPlan`
   não carrega esse dado). Documentado como limitação conhecida — não é bug
   desta story.

2. **Registro é criado sempre que `dispatchedBy !== null`, mesmo quando o
   vínculo automático (Story 17.2) acaba sendo RECUSADO** (chefe não existe
   mais no registry, chefe e worker em projetos diferentes, ou chefe não é um
   adapter de IA). Interpretação explícita da AC4: a AC só exclui o caso SEM
   `dispatchedBy` ("sem vínculo detectado... nenhum registro é criado") — não
   fala do caso em que `dispatchedBy` existe mas o vínculo automático não pôde
   ser efetivado por uma regra de negócio à parte (Story 9.1/17.2). Um
   despacho ocorreu de fato nesses casos (o worker foi lançado por um chefe
   identificável) — recusar o vínculo terminal-a-terminal não desfaz esse
   fato. Como o FR63 (Story 18.5, "contador histórico no recommend") vai
   consumir este histórico pra decidir recomendação de adapter por
   categoria/adapter, um histórico que perdesse despachos com vínculo
   recusado subrepresentaria tentativas reais — dado incompleto seria pior
   que a alternativa. O registro fica no MESMO bloco `try/catch` de
   `adoptExternalSessions`, então uma falha ali (improvável — é só
   `Map.set` + `queue.push`) não impede a adoção do worker.

3. **Desfecho (`outcome`) cobre três casos terminais**: `'done'`/`'error'`
   (via `agentStatus`, listener de `registry.onEvent` no mesmo ponto que
   dispara `planTerminalLinkRouting`, Story 9.2) e `'closed'` (fechamento do
   terminal, evento `'closed'` do registry — a AC3 menciona "done, error, ou
   fecha" explicitamente). Implementados como DOIS listeners novos e
   separados (não emendados nos listeners existentes de `terminalLinkManager`)
   pra manter uma responsabilidade por listener, seguindo o estilo já
   predominante no arquivo (cada bloco tem seu próprio `try/catch` com
   prefixo de log dedicado).

4. **Sem push de evento pro renderer** (`dispatchManager.onEvent` existe mas
   não é consumido em `session-ipc.ts`). A AC5 deixa o painel de UI
   explicitamente fora de escopo ("árvore de delegação", ideia separada não
   formalizada) — adicionar um canal IPC `dispatchEvent` sem nenhum
   consumidor seria escopo não pedido (Artigo IV, No Invention). `onEvent`
   fica pronto pra quando essa UI existir, mesma superfície pública dos
   outros managers.
