## Story

**As a** desenvolvedor multi-agente,
**I want** o state store provado sob carga real e falha parcial,
**so that** a fonte de verdade seja confiável em qualquer cenário de queda.

## Acceptance Criteria

1. Teste de integração com 6 sessões reais gerando output simultâneo: eventos e estado gravados sem perda nem bloqueio do fluxo (NFR8).
2. Batch parcial que falha no meio não corrompe o store (transação atômica da WriteQueue provada por teste — tudo-ou-nada).
3. Timeline/events sob carga preservam ordenação consultável (ts + limit) e contagens consistentes (base dos relatórios 3.5).
4. Tarefas/vínculos ficam explicitamente FORA (entidades do E5) — documentado.

## Tasks / Subtasks

- [x] Task 1 — **GAP REAL (AC2):** WriteQueue perdia o batch quando `apply` lançava (fila esvaziada ANTES do apply; exceção vazava ao caminho interativo). Fix: falha → recoloca o batch na FRENTE da fila + retry agendado; após N tentativas consecutivas descarta com erro logado (contrato: perda ≤ 1 batch, app nunca trava); exceção nunca propaga ao caller de push/flush
- [x] Task 2 — testes unit da resiliência: falha transitória (2 falhas → sucesso) recupera sem perda nem duplicação e preserva ORDEM; falha permanente descarta após cap sem travar novas gravações; contrato documentado: retry pressupõe apply transacional (SQLite applyBatch — rollback total)
- [x] Task 3 — teste de carga (AC1): 6 PTYs reais + SessionRegistry + PersistenceManager — output fluindo, eventos de domínio gravados sem perda (created×6 + closed×6 + instruction×6 na trilha), queue drena, interação não bloqueia
- [x] Task 4 — consistência sob carga (AC3): timeline ordenada (ts desc) e respeitando limit; countEvents bate com a contagem filtrada por terminal; relatório 3.5 íntegro sobre a sessão de carga
- [x] Task 5 — AC4 documentado (Dev Notes): tarefas/vínculos são entidades do E5; store atual já consolida layout/terminais/adapters/workspaces/eventos/scrollback (1.4+2.1+3.x)

## Dev Notes

- Retry é seguro PORQUE o applyBatch de produção é transacional (SQLite `db.transaction` — rollback total em falha); MemoryStateStore não simula rollback, então os testes de falha usam apply fake atômico (lança ANTES de aplicar).
- Consolidação do AC1 original do PRD já entregue: layout+terminais+adapter_id+workspace (SQLite v3), scrollback (arquivos 1.4), timeline/events (3.3), status transiente por decisão (2.1). Tarefas/vínculos → E5.
- Gotcha de ambiente que motiva o retry: antivírus causa locks transitórios em disco nesta máquina (mesma classe do retry de git add).

### Completion Notes List / File List

- **Bug real achado:** WriteQueue não tinha nenhuma proteção contra `apply` lançando — o batch já tinha sido removido da fila (`this.ops = []`) ANTES da chamada, então uma falha destruía as operações silenciosamente e ainda propagava a exceção ao caller síncrono de `flush()`/timer. Corrigido com retry-na-frente + cap + swallow.
- Retry pressupõe `apply` transacional (SQLite `db.transaction` faz rollback total) — documentado no JSDoc da classe; MemoryStateStore não simula rollback parcial (não é o alvo do teste).
- Teste de carga reusa o padrão de PTYs reais da 1.3/1.4 (mesmo spawnReal), mas acopla PersistenceManager + WriteQueue real para provar a trilha sob throughput real.
