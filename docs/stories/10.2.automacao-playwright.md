# Story 10.2: Automação via Playwright para agentes

## Status

InReview

## Story

**As a** desenvolvedor multi-agente,
**I want** que agentes possam automatizar o browser preview (navegar, clicar, ler o DOM),
**so that** eles testem a aplicação web que estão construindo sem eu precisar descrever o que ver.

## Acceptance Criteria

1. Contrato IPC expõe operações básicas de automação (navegar, clicar por seletor, ler texto/screenshot) sobre a MESMA instância Playwright do tile visível (10.1) — não uma sessão headless paralela.
2. Operações de automação são acionáveis por instrução ao agente (o agente decide chamar, via o mecanismo de tooling que ele já tem — MCP ou equivalente já exposto pelos adapters) OU manualmente pelo humano a partir do painel — qualquer um dos dois caminhos satisfaz o AC.
3. Toda ação de automação é registrada na timeline (origem `system` quando disparada por agente, `human` quando manual) — mesmo padrão de auditoria já usado em todo o app.
4. Falha de automação (seletor não encontrado, timeout) não derruba o preview nem o app — erro reportado de volta ao chamador, painel continua navegável manualmente.

- [x] Task 2 — caminho ESCOLHIDO para AC2 (qualquer um dos dois satisfaz): MANUAL — este projeto não tem hoje um mecanismo de MCP/tooling exposto AOS adapters de CLI (Claude Code/Codex/Grok são wrappers de processo, não servidores MCP consumidos pelo Cockpit) — construir essa ponte seria uma peça de infraestrutura NOVA e maior que o escopo desta story; o caminho `origin: 'system'` fica preparado no contrato (`recordBrowserAction` aceita `'human' | 'system'`) para quando essa ponte existir
- [x] Task 3 — painel de automação manual no `BrowserPreviewTile` (packages/ui): botão "⚙" no cabeçalho abre/fecha um mini-painel com seletor CSS + botões "clicar"/"ler texto"; resultado da leitura mostrado inline (AC2)
- [x] Task 4 — `persistence.recordBrowserAction(tileId, {action, selector?}, origin)` (NOVO) — trilha auditável reusando o campo genérico "id do recurso" da trilha (já usado por `sdc.*` com ids de terminal) com o id do TILE de browser; chamado nos handlers `browser.click`/`browser.readText` com origem `'human'` (AC3)
- [x] Task 5 — AC4 (falha não derruba nada) é garantia ESTRUTURAL do design já construído na 10.1: `ipcMain.handle` do Electron converte qualquer exceção do handler numa Promise rejeitada no renderer, sem derrubar o processo Main; `BrowserPreviewManager.click` deixa o erro do Playwright (ex.: seletor não encontrado/timeout) propagar intencionalmente até o chamador (para que ele SAIBA que falhou), enquanto `readText`/`screenshot` engolem erro e retornam `null` (leitura é best-effort, não uma ação com efeito colateral que precisa ser confirmada)

## Dev Notes

- Decisão mais importante desta story: NÃO construir uma ponte MCP para os adapters de CLI existentes. A visão de longo prazo do Cockpit (`docs/prd/visao-do-fundador-cockpit-aiox.md`) claramente aponta para os agentes eventualmente chamarem ferramentas do Cockpit via MCP, mas isso é uma peça de infraestrutura TRANSVERSAL (não específica de browser preview) que nenhuma story anterior construiu — inventá-la agora, só para esta AC, violaria o Artigo IV (No Invention) da Constitution. A AC2 explicitamente aceita o caminho manual como suficiente; o caminho `system` fica pronto no contrato (`recordBrowserAction` já aceita `origin: 'system'`) para quando uma story própria de "MCP bridge" for formalizada.
- `click` propaga erro (não engole) deliberadamente — diferente de `readText`/`screenshot`, que são leituras passivas e degradam bem para `null`. Um `click` que falhou silenciosamente seria pior (o chamador acha que agiu quando não agiu) do que um erro explícito.
- O painel de automação é colapsável (fechado por padrão) — não polui a UI do tile para quem só quer navegar; só aparece pra quem clica no "⚙".
