# Story 12.4: Identidade visual por adapter + adapters Gemini CLI e Antigravity

## Status

InReview

## Story

**As a** desenvolvedor multi-agente,
**I want** reconhecer visualmente qual agente (Claude, Codex, Grok, Gemini, Antigravity) está em cada terminal,
**so that** eu não precise ler o nome do adapter pra saber quem é quem.

## Acceptance Criteria

1. Cada adapter tem uma cor de identidade própria, exibida no cabeçalho do `TerminalTile` e na sessão master (mesmo princípio de token de cor já usado pra status, `STATUS_COLORS`) — tabela fixa por `adapterId`, nenhuma configuração de usuário nesta story.
2. NOVO adapter `gemini-cli` — spawna o comando `gemini`, segue o mesmo contrato normalizado (FR3): ciclo de vida do processo, entrada/saída, detecção de status (idle/working/waiting-input/done/error).
3. NOVO adapter `antigravity` — spawna o comando `agy`, mesmo contrato normalizado; instalado via `curl -fsSL https://antigravity.google/cli/install.cmd`, binário local do usuário — o adapter só invoca o comando já instalado no PATH, não faz a instalação.
4. Seletor de adapter (App.tsx, já existente desde o Épico 2) passa a listar os 6 adapters (shell, claude-code, codex, grok, gemini-cli, antigravity).

## Dev Notes

- `GeminiCliAdapter`/`AntigravityAdapter` são cópias estruturais deliberadas do `GrokAdapter` (Story 2.4) — mesma limitação documentada (CLIs REPL de terceiros sem hooks/notify de ciclo de vida expostos), então a mesma heurística validada (`\r` no write → `working`; exit code → `done`/`error`) se aplica igualmente. Nenhuma abstração nova extraída (ex.: uma "base class" compartilhada) porque os 3 adapters (grok, gemini-cli, antigravity) já são pequenos o bastante e a duplicação é mais barata que uma abstração prematura nesta fase — mesmo critério de "3 linhas parecidas > abstração cedo demais" já seguido no resto do projeto.
- `AntigravityAdapter.detectAvailability` documenta explicitamente (AC3, comentário no código) que o adapter NUNCA roda `install.cmd` — só verifica se `agy` já está no PATH do usuário. Decisão de escopo consistente com o resto do sistema de adapters: o Cockpit orquestra CLIs já instalados, nunca instala nada por conta própria (mesmo princípio do NFR6 — auth/instalação ficam com o usuário).
- Cores de identidade escolhidas (AC1): `claude-code` terracota (`#D97757`, aproxima a identidade visual da Anthropic), `codex` verde (`#10A37F`, aproxima OpenAI), `grok` violeta (`#8B5CF6`, sem cor de marca forte e universalmente reconhecida, então uma cor distinta das demais), `gemini-cli` azul (`#4285F4`, "Google Blue"), `antigravity` amarelo (`#F4B400`, também da paleta Google, mas distinto do azul do Gemini pra diferenciar os dois produtos do mesmo fabricante), `shell`/`cmd` cinza neutro (já era a cor "sem identidade" implícita). Nenhuma dessas cores é configurável pelo usuário nesta story (AC1 é explícito sobre isso).
- Seletor de adapter (AC4) não precisou de NENHUMA linha de código nova — já era escrito de forma genérica desde o Épico 2 (`adapters.map(...)`), reagindo à lista real do registry. Registrar os adapters no host já basta; a UI reflete automaticamente. Bom sinal de que o contrato de adapter (Épico 2) foi bem desenhado desde o início.
