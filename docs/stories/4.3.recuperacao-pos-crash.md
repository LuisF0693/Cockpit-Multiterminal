# Story 4.3: Recuperação pós-crash com resumo (revisada pós-daemon)

## Status

InReview

## Story

**As a** desenvolvedor multi-agente,
**I want** que após um crash o cockpit me mostre o que estava em andamento,
**so that** eu retome com confiança em vez de reconstruir a situação de memória.

## Acceptance Criteria

1. Crash detectado na reabertura (flag `clean_shutdown`, FR12 — já existe desde a 1.4); boot NÃO relança/adota automaticamente enquanto a tela de recuperação não for resolvida.
2. Tela de recuperação exibe: terminais ativos no momento do crash (nome, adapter, cwd, último status conhecido pela trilha) e últimos eventos globais da timeline. "Tarefas em andamento" fica FORA (entidade do E5, ainda não existe).
3. Usuário escolhe: restaurar tudo, restaurar seletivamente (checklist) ou iniciar sessão limpa — terminais não escolhidos são ARQUIVADOS (nunca destruídos, mesma garantia da 1.4).
4. Evento de recuperação (`crash.recovery`: escolha + contadores) registrado na timeline.

## Dev Notes

- Reuso deliberado: adoção+restore da recuperação é a MESMA função interna já usada no boot automático (`performResume`) — a única diferença é QUANDO ela roda e se algum terminal foi pré-arquivado antes.
- "Decisões pendentes" (texto original do PRD) não é reconstruível no boot: `agentStatus` é transiente (Dev Notes 2.1) — não existe status ao vivo antes de retomar a sessão. Substituído por "último status conhecido" via `status.changed` (3.5), que é o dado real disponível.
- Boot em modo daemon: se todas as sessões estavam vivas no daemon (crash só derrubou o Main), a recuperação ainda passa pela tela — consistência > atalho (o usuário decide mesmo quando a adoção salvaria tudo de graça).
- `markBootStart()` muda de lugar (para dentro de `registerSessionIpc`) sem mudar semântica — só antecipa a leitura para poder computar `crashDetected` antes do primeiro IPC.

### Completion Notes List / File List

- **Bug real achado durante o desenvolvimento:** o teste de integração da 6.4 ("attach com tailBytes 0 resubscreve SEM replay") tinha uma condição de corrida — o cliente A desconectava assim que via o eco DIGITADO do comando (echo local do terminal), antes da EXECUÇÃO real do `echo` completar; o resultado do comando podia então chegar como stream VIVO ao cliente B logo após o attach, sendo confundido com replay indevido. Corrigido fazendo o teste aguardar um "settle marker" (segundo comando) antes de desconectar — prova que o shell voltou ao prompt e não há mais bytes pendentes. Bug do TESTE, não do daemon (a lógica de `readScrollbackTail` com `maxBytes=0` sempre retorna 0 bytes, confirmado por análise).
- `markBootStart()` migrou de `index.ts` para dentro de `registerSessionIpc` — precisa resolver `crashDetected` ANTES do primeiro IPC ser registrado; chamada duplicada removida do Main.
- `performResume` é o ÚNICO caminho de adoção+relaunch — usado tanto pelo boot automático quanto pela resolução da Recovery Screen; nenhuma lógica duplicada.
- Atalhos de teclado e criação de terminal ficam desabilitados durante `view === 'recovery'` — evita criar sessões "soltas" antes da decisão do usuário.
- **Criados:** `packages/ui/src/recovery-screen.tsx`. **Modificados:** shared (`ipc.ts`: CrashSummary/RecoveryResolve schemas+canais+CockpitApi), core (`persistence.ts` crashSummary/archiveForCrash/recordCrashRecovery, `persistence.test.ts`), Main (`session-ipc.ts` crashDetected+performResume+handlers, `index.ts` boot gateado), preload, ui (`index.ts`), App.tsx (view recovery + seedFromMain/resolveRecovery), pty-host (`daemon-lifecycle.integration.test.ts` fix de race).
