# Story 5.3: Pontos de decisão humana auditáveis

## Status

InReview

## Story

**As a** desenvolvedor multi-agente,
**I want** aprovar, rejeitar ou redirecionar o trabalho em pontos de decisão explícitos,
**so that** a governança faça parte do fluxo com trilha auditável completa.

## Acceptance Criteria

1. Tarefa em "aguardando decisão" expõe ações: aprovar (→ revisada/concluída), rejeitar (→ em execução com feedback) e redirecionar (→ outro agente).
2. Cada decisão registra autor, timestamp e justificativa opcional na timeline (FR15).
3. Decisões pendentes de tarefas integram a fila de decisões do master (unificada com waiting-input dos agentes).
4. Redirecionamento transfere o vínculo e notifica o novo agente via instrução inicial.

## Dev Notes

- Redirect é uma operação de DUAS partes que cruza responsabilidades: `TaskManager.decide()` (estado da tarefa) + `registry.linkTask()` (vínculo do terminal) — só o Main (session-ipc.ts) enxerga os dois, então a orquestração mora lá, não dentro do TaskManager.
- Vínculo lido de `registry.list()` (fonte viva), não do store — consistente com o padrão já usado em `instructTaskAgents` (5.2).
- Instrução de redirect é enviada pelo RENDERER (App.tsx via `instructAgent`), nunca pelo Main — decisão crítica 4 (dados de terminal só atravessam a MessagePort do renderer).
- "Aprovar" NUNCA pula direto para `done`: o AC menciona "revisada/concluída" como o destino eventual do fluxo, não uma transição nova; `reviewed → done` já é self-service via TasksPanel.
