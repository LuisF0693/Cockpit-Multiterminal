# Story 5.2: Vínculo tarefa ↔ agente ↔ terminal

## Status

InReview

## Story

**As a** desenvolvedor multi-agente,
**I want** vincular tarefas aos agentes/terminais que as executam,
**so that** eu sempre saiba quem está fazendo o quê e por quê.

## Acceptance Criteria

1. Tarefa pode ser vinculada/desvinculada a um ou mais terminais/agentes.
2. Vínculo visível nos dois sentidos: no tile/foco do terminal e no card da tarefa; dashboard master exibe a tarefa de cada agente (completa FR6).
3. Enviar instrução a partir de uma tarefa direciona ao(s) agente(s) vinculado(s).

- [x] Task 2 — `StateStore.setTerminalTask(id, taskId)` (sqlite+memory); `SessionRegistry.linkTask(id, taskId|null)` emite evento `task_linked` (mesmo padrão do `renamed`/`status`); `PersistenceManager.wire()` trata `task_linked` → upsert do terminal (persiste, inclusive no relaunch do `restore()` e na adoção) + trilha `terminal.task_linked` com payload `{taskId}`
- [x] Task 3 — contrato: `taskId` no `SessionRecordSchema`; `TaskLinkRequestSchema {terminalId, taskId: string|null}`; canal `session.linkTask`; `CockpitApi.session.linkTask`
- [x] Task 4 — IPC (Main) + preload
- [x] Task 5 — UI (AC2): master dashboard troca a coluna "tarefa: —" fixa por dropdown real (via lookup tasks[], vincula/desvincula direto); `TasksPanel` (card da tarefa) lista terminais vinculados (derivado de `sessions.filter(s => s.taskId === task.id)` — sem round-trip extra) com botão de desvincular
- [x] Task 6 — UI (AC3): campo de instrução no card da tarefa (TasksPanel) que envia a MESMA instrução para todos os terminais vinculados, reusando `instructAgent` do App.tsx (já existente da 3.2) — sem lógica de porta duplicada
- [x] Task 7 — testes unit: registry.linkTask emite evento + persiste + trilha; preserva demais campos; vínculo sobrevive ao relaunch clássico do restore(); verify suíte completa PASS (122) + smoke:daemon PASS

## Dev Notes

- Um terminal → no máximo uma tarefa é a simplificação deliberada: evita tabela de junção agora; se "um terminal em duas tarefas simultâneas" virar requisito real, revisitar na 5.4 (Board).
- `taskId: null` desvincula — mesmo canal `session.linkTask` cobre vincular e desvincular (payload com `taskId: null`).
- Instrução a partir da tarefa reusa `instructAgent(id, text)` do App.tsx por terminal vinculado — nenhuma nova infraestrutura de porta binária.
- Vínculo sobrevive a restart/adoção (persistido na coluna `task_id`) — consistente com o restante do state store.
