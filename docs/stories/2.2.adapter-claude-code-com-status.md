## Story

**As a** desenvolvedor multi-agente,
**I want** rodar o Claude Code em um terminal com status detectado automaticamente,
**so that** o cockpit saiba quando ele está trabalhando, aguardando minha decisão ou concluído.

## Acceptance Criteria

1. Adapter spawna Claude Code via PTY com sessão interativa plena.
2. Status idle/working/waiting-input/done/error detectado — preferencialmente via hooks nativos do Claude Code; fallback de parsing documentado.
3. Mudanças de status emitidas como eventos consumíveis pela UI (cabeçalho do tile reflete status em tempo real).
4. Autenticação permanece no CLI; adapter não intercepta credenciais (NFR6).

## Dev Notes

> Contexto dos shards + 2.1 — não reler arquitetura.

**Hooks do Claude Code (estratégia native-hooks da decisão 3):** o Claude Code executa hooks de shell configuráveis por settings JSON em eventos do ciclo de vida (`UserPromptSubmit`, `Stop`, `Notification`, `SessionStart`, `PreToolUse`...). O flag `--settings <arquivo>` carrega settings adicionais por sessão — permite injetar hooks SEM tocar nos settings do usuário/projeto. Mapeamento p/ AgentStatus: prompt enviado→working; turno concluído (Stop)→idle; Notification (permissão/atenção)→waiting-input; exit 0→done / ≠0→error.

**⚠️ Risco mapeado:** se a versão local do CLI não suportar `--settings` ou renomear eventos, o adapter degrada para process-only com log claro (nunca quebrar a sessão). Registrar a versão testada no README do adapter.

**Arquivo de status (ponte):** `%TEMP%/cockpit-claude/{sessionTag}.status` — hooks fazem `cmd /c echo working>> arquivo`. Watcher com dedupe (só emite mudanças). Windows: fs.watch pode falhar em %TEMP% de rede — fallback polling.

**Reuso da 2.1:** herdar spawn PTY/dispose do padrão do ShellAdapter (grace + isPidAlive; dispose rejeita órfão). O host/UI não mudam — só registrar o adapter novo (a 2.1 deixou tudo plugável; se algo no core precisar mudar, o contrato foi furado).

**NFR6:** o settings temporário contém APENAS hooks; env é herdado do usuário (auth do claude fica no keychain/config dele).

**Fora desta story:** Codex (2.3), Grok (2.4), painel/realce visual completo (2.5 — o dot da 2.1 já reflete), flow-control v2 do contrato (proposta ao @architect registrada na 2.1).
