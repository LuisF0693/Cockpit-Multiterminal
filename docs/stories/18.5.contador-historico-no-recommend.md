# Story 18.5: Contador histórico no `--recommend`

## Status

Done

## Story

**As a** chefe avaliando candidatos via `--recommend`,
**I want** ver quantas vezes cada adapter foi aprovado/rejeitado historicamente naquela categoria,
**so that** minha decisão de qual adapter escolher seja embasada em dado real, não só na matriz estática editada à mão.

## Acceptance Criteria

1. `--recommend` inclui, por candidato, uma contagem agregada do histórico do FR62 (Story 18.4): desfechos `done` vs `error` por `adapterId`.
   > **Correção de escopo (2026-07-20, Dex):** a AC original pedia a contagem "por adapter + categoria". Investigando a implementação real da 18.4, `PersistedDispatchRecord`/`DispatchRecord` (`packages/core/src/state-store/types.ts`, `packages/core/src/dispatch-manager.ts`) NÃO guarda a categoria da tarefa (`development`/`review-planning`/etc.) — só `adapterId`, `projectId`, `model`, `dispatchedBy`, `workerId`, `createdAt`, `outcome`. A categoria é decidida em `planAgentDispatch` (CLI, no momento do despacho — `agent-dispatch.ts`) mas não atravessa o protocolo do daemon até o ponto onde o registro nasce, na adoção externa (`ExternalSessionInfo`/`ExternalAdoptionPlan`, `external-adoption.ts`, também sem esse campo). Plumbar categoria por todo esse caminho reabriria escopo do Épico 17/18.4 — fora do escopo desta story. **Decisão: a agregação é SÓ POR ADAPTER**, sem quebra por categoria. Ver Dev Agent Record.
2. Puramente informativo — a ORDEM dos candidatos continua vindo só da matriz de capacidades (Épico 17.2); o contador é anexado à `reason` de `explainCandidates`, nunca reordena.
3. Sem histórico (app novo, ou adapter nunca despachado), o campo aparece vazio/zerado — `--recommend` nunca quebra por falta de dado.
4. Depende da Story 18.4 (histórico precisa existir pra ter o que contar).

## Dev Agent Record

### Decisão de arquitetura: como a CLI acessa o histórico do Main (a parte crítica desta story)

A story pedia pra avaliar duas opções — (a) SQLite direto, (b) novo pipe command no daemon — e documentar a escolha.

**Investigação:**
- `DispatchManager` (fonte do histórico, Story 18.4) vive inteiramente no processo **Main** (Electron), instanciado em `session-ipc.ts`. A CLI `agent-dispatch` roda como **processo Node puro separado** (`node out/main/agent-dispatch.js` — ver `package.json` script `agent:dispatch`), sem nenhum vínculo de processo com o Main.
- **Opção (a) — SQLite direto:** inviável na prática. `packages/core/src/state-store/sqlite-state-store.ts` documenta explicitamente (comentário original, decisão crítica 2 da Story 1.4): "o driver é INJETADO pelo Main: better-sqlite3 é rebuildado para a ABI do Electron e não carrega sob Node". `apps/desktop/src/main/index.ts` importa `better-sqlite3` e injeta a instância no `SqliteStateStore` — só o Main tem esse binário nativo compatível. A CLI rodando sob `node` puro (confirmado: ABI diferente do Electron) não conseguiria abrir o mesmo `.db`, exigiria uma SEGUNDA build nativa do better-sqlite3 (uma pra ABI Node, outra pra ABI Electron) só pra esta story — risco e complexidade desproporcionais a "mostrar um contador informativo".
- **Opção (b) — novo comando no protocolo do daemon:** o texto da story alertava corretamente que o daemon (`pty-host`, processo `DaemonServer`) e o Main são processos DIFERENTES, e que o `DaemonServer` não tem NENHUM acesso a `StateStore`/SQLite — ele é puramente in-memory (`Map<string, DaemonSession>`, sem persistência própria). Um comando `dispatch-history` ingênuo no daemon não teria de onde ler.
- **A peça que destrava:** `apps/desktop/src/main/daemon-manager.ts` (`DaemonManager`, Story 6.3) mostra que o **Main já é cliente do próprio daemon** — ele fala com o `DaemonServer` pelo MESMO named pipe que a CLI usa, via `DaemonClient` (`packages/pty-host/src/daemon-client.ts`), para orquestrar PTYs. Isso significa que o Main pode **empurrar** dados pro daemon usando a conexão que já existe, sem abrir canal novo.

**Decisão implementada:** o daemon vira um **relay/cache em memória, sem persistência própria** — exatamente como já é pra sessões PTY (`this.sessions`, um `Map` que também não sobrevive a restart do daemon). Dois novos comandos no protocolo (`packages/pty-host/src/daemon-protocol.ts`, mesmo padrão de `list-sessions`):
- `dispatch-history-push` (inbound, **fire-and-forget, sem requestId/ack**): o Main empurra o snapshot mais recente (`aggregateDispatchOutcomes(dispatchManager.list())`, `@cockpit/core`) toda vez que o `DispatchManager` muda (`dispatchManager.onEvent(...)`) e uma vez no boot (snapshot inicial, pro histórico carregado do SQLite já ficar disponível). Sem ack porque é sempre um snapshot COMPLETO — se um push se perder, o próximo evento reenvia tudo; não há necessidade de merge incremental nem de garantir entrega.
- `dispatch-history` (inbound, request/response com `requestId`, mesmo padrão de `list-sessions`/`ping`): a CLI consulta o cache. Devolve `[]` se o Main nunca conectou/empurrou (app nunca foi aberto desde o boot do daemon) — nunca lança.

`DaemonServer.dispatchHistoryCache: AdapterOutcomeCount[]` é o único estado novo no daemon — substituído inteiro a cada push, sem merge/dedupe (mais simples que rastrear por id). `PtyBackend.pushDispatchHistory?` é OPCIONAL na interface (`session-ipc.ts`) — o backend `utilityProcess` clássico (`COCKPIT_NO_DAEMON=1`) não implementa (não tem CLI externa persistente pra servir; a chamada vira no-op via optional chaining).

**Por que isso não é "reabrir escopo do 17/18.4":** nenhuma mensagem nova carrega categoria — só `{adapterId, done, error}`, o dado que a 18.4 já persiste e a 18.5 pediu para expor. O daemon não ganhou nenhuma responsabilidade de persistência; ele segue sendo hospedeiro de PTYs com um cache efêmero a mais, do mesmo jeito que já cacheia `label`/`dispatchedBy` de sessões vivas.

### Outras notas de implementação

- `aggregateDispatchOutcomes` (core) só conta `done`/`error` como desfecho conclusivo — `closed` (fechamento sem status terminal explícito, Story 18.4 AC3) e `null` (ainda em andamento) ficam de fora: nenhum dos dois indica se o adapter "funcionou" ou "falhou".
- `explainCandidates` ganhou um 3º parâmetro `outcomeCounts?: readonly AdapterOutcomeCounts[]` — opcional, no FINAL da assinatura, então nenhum call site existente (`agent-dispatch.ts`, testes da 17.2/17.3) precisou mudar. O sufixo `— histórico: N concluído(s), M erro(s)` só aparece quando `done > 0 || error > 0` pro adapter em questão (AC3).
- Smoke real não executado (ambiente sem app Electron rodando neste worktree); a integração ponta a ponta é coberta pelo teste `daemon.integration.test.ts` novo, que sobe um `DaemonServer` real e simula dois clientes (Main empurrando, CLI consultando) no mesmo pipe.
- `pnpm install` foi necessário no worktree (sem `node_modules` — worktree novo) antes de rodar typecheck/testes.
