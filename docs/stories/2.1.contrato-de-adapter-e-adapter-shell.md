## Story

**As a** desenvolvedor multi-agente,
**I want** que cada terminal seja hospedado por um adapter com contrato único,
**so that** qualquer CLI presente ou futuro possa ser integrado sem tocar no core.

## Acceptance Criteria

1. Contrato de adapter definido e documentado: spawn, ciclo de vida, entrada/saída, sinais de status, encerramento.
2. Adapter genérico de shell implementa o contrato (status básico: running/exited).
3. Core não referencia nenhum provider específico (NFR7) — verificado por teste/lint de dependência.
4. Criar terminal permite escolher o adapter; documentação de como escrever um novo adapter em `docs/guides/`.

## Dev Notes

> Contexto dos shards — não reler arquitetura.

**Contrato é LEI (decisão crítica 3):** implementar EXATAMENTE a interface documentada — `AgentAdapter` (id, displayName, statusStrategy, detectAvailability, spawn) e `AgentSession` (write, resize, dispose, onData, onStatus, onExit). Divergências exigem consulta ao @architect.

**Fronteira de processos (components.md):** adapters vivem NO PTY HOST (utilityProcess). O core (Main) nunca vê um adapter — só o registry via mensagens do protocolo. A regra ESLint que restringe `**/adapters/*` já existe no eslint.config.mjs (1.1) — estender e criar o teste de dependência.

**Refactor conservador do host:** o `PtySessionManager` da 1.2 vira a BASE do ShellAdapter (ou é consumido por ele) — não jogar fora o código validado; o wiring de MessagePort/backpressure/scrollback (1.4) permanece intocado no host-entry. O que muda: quem spawna é o adapter, e o host emite `session-status` além de `session-exit`.

**AgentStatus (data-models.md):** 'idle' | 'working' | 'waiting-input' | 'done' | 'error'. Shell (process-only): vivo → 'working'; exit 0 → 'done'; exit ≠ 0 → 'error'. Os status ricos chegam nas 2.2+ (native-hooks/output-parsing).

**Persistência (1.4):** nova coluna `adapter_id TEXT NOT NULL DEFAULT 'shell'` — migração incremental guardada por `app_meta.schema_version` (1 → 2). Restore relança com o adapter salvo. `agentStatus` NÃO persiste (transiente — recalculado no spawn).

**NFR6 (credenciais):** spawn herda `process.env` do usuário; adapter nunca recebe/loga tokens. Nada de env custom nesta story.

**UI mínima (AC4):** o seletor pode ser um dropdown no botão "+ novo terminal" (header) listando adapters disponíveis via novo canal `adapter.list` (Zod). Com só o shell, a UX é trivial — o ponto é o CONTRATO da criação já aceitar adapterId.

**Fora desta story:** adapters Claude/Codex/Grok (2.2-2.4), painel de status colorido (2.5 — mas o campo `agentStatus` no SessionRecord + evento já nascem aqui), hooks nativos.
