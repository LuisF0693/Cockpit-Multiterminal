# Story 17.3: Modelo por sessão no despacho de workers

## Status

InReview

## Story

**As a** chefe que despacha workers pelo Cockpit,
**I want** que o worker já nasça com o MODELO escolhido (não só a CLI) — Haiku pra tarefa barata, Sonnet pra média, GPT-5.x no codex, Gemini na pesquisa,
**so that** o custo e a força de cada despacho sejam calibrados por tarefa, sem trocar modelo na mão depois que o terminal abre.

## Acceptance Criteria

1. Todos os adapters de IA (claude-code, codex, gemini-cli, grok, antigravity) repassam `config.args` ao spawn da CLI (o contrato e o daemon já propagavam desde a 12.6 — só o Ollama consumia).
2. A CLI `agent-dispatch` aceita `--model <nome>`; o worker nasce com `--model` na linha de comando da CLI alvo (Ollama: `run <modelo>`). `--model` exige `--adapter` explícito — a grafia do modelo é específica de cada CLI, então quem escolhe modelo escolhe a CLI.
3. A matriz de capacidades ganha `models` (sugestões por adapter, editáveis em `adapters-profile.json`) e o `--recommend` os exibe na justificativa.
4. Testes cobrem o repasse de args nos adapters e os modelos na justificativa.
5. Docs atualizadas (AGENTS.md) e gates verdes.

## Tasks / Subtasks

- [x] Task 1 — Adapters repassam `config.args` (AC: 1, 4).
- [x] Task 2 — CLI `--model` com validação e mapeamento por adapter (AC: 2).
- [x] Task 3 — `models` na matriz + `--recommend` (AC: 3, 4).
- [x] Task 4 — Docs + gates + smoke (AC: 5).

## Dev Notes

- Troca de modelo em sessão VIVA continua manual (digitar `/model` no claude/codex) — o chefe pode enviá-la via bridge/vínculo; automatizar isso é story futura se a dor aparecer.
- Os nomes em `models` do `adapters-profile.json` são dicas EDITÁVEIS do fundador, na grafia que cada CLI aceita; a CLI não valida o nome (erro de modelo aparece no terminal do worker).
