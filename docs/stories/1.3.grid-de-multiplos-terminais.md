## Story

**As a** desenvolvedor multi-agente,
**I want** criar, nomear, redimensionar e fechar vários terminais em um grid,
**so that** eu opere múltiplos CLIs lado a lado em um único app.

## Acceptance Criteria

1. Usuário cria novos terminais (botão e atalho), cada um com PTY independente.
2. Terminais podem ser nomeados/renomeados e exibem cabeçalho com nome.
3. Grid permite reorganizar e redimensionar tiles; ≥ 6 terminais simultâneos estáveis e interativos (NFR3).
4. Fechar tile individual não afeta os demais.
5. Atalhos de teclado para alternar foco entre terminais.

## Dev Notes

> Contexto dos shards + decisões do fundador — não reler arquitetura.

**⛔ Dependência dura: Story 1.2 concluída.** Esta story MULTIPLICA o que a 1.2 cria (TerminalView + canal binário MessagePort + pty-host). Cada novo terminal = nova sessão no pty-host + novo par de MessagePorts negociado pelo Main. Não redesenhar esse fluxo — replicá-lo por sessão.

**Direção de design do fundador (front-end-spec.md → Design References):** o "grid" é um **canvas com liberdade de arranjo** — tiles móveis, tamanhos livres, snap opcional — como nas capturas em `docs/design-references/`. NÃO implementar grid rígido de células. Sidebar é **árvore de navegação** (sessões nesta story), colapsável, mais larga que rail de ícones.

**Session Manager (docs/architecture/components.md):** vive em `packages/core` (Main process). É a fonte de verdade do registro de sessões; a UI reflete eventos, nunca mantém estado próprio de sessões (CQRS leve — coding standards). O pty-host executa; o core registra e coordena.

**Modelo de layout serializável (preparação 1.4):** estrutura pura JSON-friendly (sem classes/funções), com versão de schema (`layoutVersion: 1`). A 1.4 vai gravá-lo no state store — aqui ele vive em memória (Zustand no renderer, espelho no core via eventos).

**Performance (NFR3 + front-end spec):** ≥ 6 xterm ativos → WebGL addon já vem da 1.2; adicionar throttle de render para tiles sem foco; drag/resize nunca pode bloquear o input do terminal focado. Números de referência do spike: 6 PTYs ≈ 141MB RSS estáveis.

**Interações do TerminalTile (front-end spec):** cabeçalho sempre visível (nome, futuro status/adapter — placeholder de status entra no E2); drag pelo cabeçalho; `Ctrl+Enter` expande (Focus View completo é do E3 — aqui só maximizar/restaurar simples se trivial, senão deixar de fora).

**Fora desta story:** persistência de layout (1.4), adapters/status coloridos (E2), Focus View com contexto (E3), Dashboard master (E3).

### Completion Notes List

- **Arquitetura CQRS leve:** `SessionRegistry` (@cockpit/core, Main) é a fonte de verdade; renderer espelha eventos `session.event` num store Zustand. A UI nunca é dona do estado de sessões.
- **Fluxo da 1.2 replicado por sessão (não redesenhado):** cada create = nova sessão no pty-host + novo par de MessagePorts negociado pelo Main, porta tagueada com o session id.
- **Canvas livre conforme Design References:** posições/tamanhos livres, snap 8px no fim do gesto (drag fluido), zIndex por bringToFront, cascata para novos tiles.
- **Performance:** tiles desfocados escrevem em lote (~10fps) preservando acks (backpressure); focado escreve imediato; ULID próprio (25 linhas) evitou dependência.
- **Host resiliente:** exit espontâneo do shell → evento `exited` (tile marca "encerrado"); crash do host → todas marcadas exited + respawn automático.
- **Testes: 31 no total** — 7 registry + 1 integração multi-sessão (6 PTYs reais, AC3/AC4) + 6 layout + 4 atalhos + 9 pty-host + 4 shared.
