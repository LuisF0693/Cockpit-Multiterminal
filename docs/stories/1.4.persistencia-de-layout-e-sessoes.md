## Story

**As a** desenvolvedor multi-agente,
**I want** que o app reabra com o mesmo layout e terminais da última sessão,
**so that** eu não reconstrua meu ambiente a cada restart.

## Acceptance Criteria

1. Layout do grid (quantidade, nomes, posições, tamanhos dos terminais) persiste localmente de forma contínua e assíncrona (NFR8).
2. Ao reabrir o app, o layout é restaurado e os terminais são relançados nos mesmos diretórios de trabalho.
3. Scrollback de cada terminal é persistido e restaurado (limite configurável).
4. Teste de integração automatizado cobre o ciclo persistir → fechar → restaurar.

## Dev Notes

> Contexto dos shards — não reler arquitetura.

**Decisão crítica 2 (persistência):** SQLite better-sqlite3 WAL como verdade estrutural + event-log; scrollback FORA do DB em arquivos append por terminal. Write queue: input nunca espera I/O; no pior crash perde-se ≤ 250ms de batch. Boot lê estado direto (sem replay de eventos — NFR4 < 10s).

**Gotcha de módulo nativo (herdado da 1.2):** node-pty escapou do rebuild por ter prebuilds N-API; better-sqlite3 usa V8 API — EXIGE rebuild para a ABI do Electron (`@electron/rebuild`, já devDependency). Após o rebuild, better-sqlite3 não carrega mais sob Node puro: NUNCA importar better-sqlite3 em testes vitest — usar a interface `StateStore` com fake. O ciclo real é validado por script Electron-run (padrão `smoke:abi`).

**Onde vive o quê:** SqliteStateStore + write queue no Main (via `packages/core` com driver injetado); ring buffer de scrollback no PTY Host (já recebe todos os chunks — zero custo extra de IPC); arquivos em `userData/state/`.

**Restore reusa o fluxo da 1.3:** recriar sessão = `SessionRegistry.create` com `cwd`/`name` salvos; o renderer recebe tudo por `session.list`/eventos — NÃO criar caminho paralelo de restore na UI. Layout restaurado entra como estado inicial do store Zustand.

**Schema (visão lógica de data-models.md):** `terminals` ganha as colunas do tile (x, y, width, height, zIndex) — o `gridPosition` do modelo conceitual. `adapterId` fica `'shell'` fixo até o E2. `events` append-only com payload JSON validado por Zod.

**Fora desta story:** Recovery Screen de crash (E5/FR12 — aqui só a flag), tasks/decisions (E3), múltiplos perfis/settings UI.
