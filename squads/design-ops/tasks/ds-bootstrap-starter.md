# Bootstrap Design Starter

> Task ID: ds-bootstrap-starter
> Owner: `design-chief`
> Version: 2.0.0
> **Execution Type:** `Hybrid`
> **Human Checkpoint:** Confirm whether the request is only technical boilerplate configuration or also requires brand strategy/logo creation
> **Dependencies:** depends_on: `[]` · enables: `[ds-build-component, ds-theme-multi-brand, ds-integrate-workspace]` · workflow: `starter-runtime`

## Description

Bootstrap, harden, or export the real `apps/aiox-design-starter` runtime.

This task is specifically for the current starter architecture that lives inside this monorepo. It covers:

- local-first brandbook/design-system runtime setup
- starter contracts in `starter/`
- variants and presets
- token sync and runtime generated files
- standalone export to another repository

This task does **not** own brand strategy, naming, logo creation, or positioning. If the user needs to invent or redefine the brand itself, hand off to `brand` and return here after the identity inputs exist.

## Canonical Context

Before executing, read:

- `apps/aiox-design-starter/README.md`
- `apps/aiox-design-starter/package.json`
- `apps/aiox-design-starter/starter/site.config.yaml`
- `apps/aiox-design-starter/starter/design-system.config.yaml`

Treat those files as the canonical contract of the current starter.

### Token Defaults Source (NEW)

When the starter needs to hydrate initial tokens (color/spacing/typography/radius/shadow/motion/breakpoints):

- **PRIMARY:** `templates/ds-tokens-defaults-tmpl.yaml` (hydrated DTCG, justified defaults)
- **JUSTIFICATION:** `data/tier-1-token-defaults-justification.yaml` (cross-DS matrix)
- **SCHEMA CONTRACT:** `templates/ds-tokens-schema-tmpl.yaml` (canonical shape)

Use these instead of guessing values or asking the operator 70 times. Only
brand-specific decisions require human input (`accent-primary`, `font-heading`).

## Output Schema

- **produces:** a configured or hardened `aiox-design-starter` source app, or a standalone exported copy
- **format:** validated Next.js runtime with contracts, generated token bridge, and passing starter quality gates
- **consumed_by:** maintainers of `apps/aiox-design-starter` or any downstream repository that receives a standalone export

## Supported Modes

### 1. In-repo source mode

Operate directly on:

```bash
apps/aiox-design-starter
```

Use this when the user wants to evolve the canonical starter.

### 2. Standalone export mode

Materialize a self-contained copy via:

```bash
npm --prefix apps/aiox-design-starter run export:standalone -- --target=<path>
```

Use this when the user wants the starter inside another repository or workspace.

## Workflow

### Step 1: Classify scope

Separate the request into one of these buckets:

- **technical starter bootstrap**: in scope here
- **brandbook/runtime hardening**: in scope here
- **variant/preset/export**: in scope here
- **brand strategy/logo/positioning/naming**: out of scope -> handoff to `brand`

### Step 2: Load canonical starter context

Read the root README and package/config files listed above.

Confirm the current runtime assumptions:

- local-first contracts live in `starter/`
- runtime sources are `starter` or `workspace`
- variants inherit base and override selectively
- `variant3` is the local preset
- standalone export is first-class
- editorials required for portability live in `src/vendor/`

### Step 3: Resolve target mode

If the user wants the canonical starter configured, stay in:

```bash
apps/aiox-design-starter
```

If the user wants another repo bootstrapped, export first:

```bash
npm --prefix apps/aiox-design-starter run export:standalone -- --target=<target>
```

### Step 4: Configure contracts

Bootstrap or harden the relevant contract layer:

| Concern | Canonical file |
|--------|-----------------|
| Branding/chrome/SEO/navigation | `starter/site.config.yaml` |
| Theme/tokens/apps binding | `starter/design-system.config.yaml` |
| Brandbook content contract | `starter/content/brandbook.yaml` |
| Variant override | `starter/variants/<slug>/...` |
| Active runtime brand preset | generated by `scripts/sync-dtcg-tokens.cjs` |

### Step 5: Sync runtime artifacts

Generate the runtime artifacts before validating:

```bash
npm --prefix apps/aiox-design-starter run tokens:sync:write
```

This updates generated files such as the token bridge and active preset module.

### Step 6: Validate source app

Mandatory checks:

```bash
npm --prefix apps/aiox-design-starter run lint
npm --prefix apps/aiox-design-starter run typecheck
npm --prefix apps/aiox-design-starter run test:tenant-runtime
```

Stronger matrix validation:

```bash
node squads/design-system/scripts/validate-design-starter-matrix.cjs
```

### Step 7: Validate target export when applicable

For standalone targets:

1. Confirm exported files exist
2. Confirm `README.md`, `package.json`, `starter/`, `src/`, `src/vendor/` are present
3. If dependencies are installed in the target repo, re-run the same starter quality gates there

## Commands That Matter Today

The current starter uses these commands:

```bash
npm run dev
npm run build
npm run lint
npm run typecheck
npm run test:tenant-runtime
npm run scaffold:variant -- --slug=<slug> --name="<name>" --force
npm run tokens:sync:write
npm run export:standalone -- --target=<path>
```

Do **not** assume legacy commands such as:

- `npm run setup`
- `npm run doctor`

Those are not part of the current starter contract.
